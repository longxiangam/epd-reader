# 第 9 篇：错误处理与调试

> 用 Rust 构建ESP32-C3 电子墨水屏阅读器 · 系列文章

## 引言

在嵌入式开发中，调试是一件痛苦的事情。没有 GDB 的时候，你只能靠 `printf` 大法；设备突然重启了，你不知道为什么；程序崩溃了，你连调用栈都看不到。

本项目的错误处理系统通过自定义 Panic Handler、Flash 错误日志、调用栈回溯三层机制，让"黑盒"设备也能提供有价值的调试信息。

## 1. 自定义 Panic Handler

Rust 在 `#![no_std]` 环境中没有默认的 panic 处理。你必须自己实现 `#[panic_handler]`：

```rust
// src/panic.rs:9-100
#[panic_handler]
fn panic_handler(info: &PanicInfo) -> ! {
    println!("!! 发生panic错误 !!");

    // 1. 构建错误信息
    let mut error_msg = heapless::String::<200>::new();

    // 文件位置
    if let Some(location) = info.location() {
        let _ = write!(error_msg, "location: {}:{}", location.file(), location.line());
    }

    // panic 消息
    let _ = write!(error_msg, "error: {}", info.message());

    // 2. 捕获调用栈
    let backtrace = Backtrace::capture();
    let mut stack_count = 0;
    for frame in backtrace.frames() {
        if stack_count < 10 {
            let _ = write!(error_msg, "  #{} 0x{:x}\n", stack_count, frame.program_counter());
            stack_count += 1;
        } else { break; }
    }

    // 3. 保存到 Flash
    let mut error_storage = ErrorLogStorage::read().unwrap_or_default();
    error_storage.error_count += 1;
    error_storage.last_error = error_msg;
    error_storage.write();

    // 4. 等待 Flash 写入完成
    for _ in 0..1000000 {
        core::hint::spin_loop();
    }

    // 5. 停止（实际设备需要手动重启）
    loop {}
}
```

### 关键设计决策

**为什么不自动重启？** 代码最后是 `loop {}` 而不是 `software_reset()`。这是故意的——如果 panic 后自动重启，可能会陷入崩溃循环（不断 panic → 重启 → panic）。让设备停在 panic 状态，可以通过串口看到完整的错误信息。

**为什么用 `spin_loop()` 等待？** Panic handler 中不能使用异步操作（`.await`），也不能使用阻塞延时（需要硬件定时器）。`spin_loop()` 是一个忙等待循环，作为 CPU 空操作提示，让 Flash 写入有时间完成。

## 2. 调用栈回溯

### 2.1 Backtrace 捕获

`esp-backtrace` crate 提供了 RISC-V 架构的栈回溯能力：

```rust
// src/panic.rs:40-48
let backtrace = Backtrace::capture();
for frame in backtrace.frames() {
    let _ = write!(error_msg, "  #{} 0x{:x}\n", stack_count, frame.program_counter());
}
```

esp-hal v1.x 使用 `Backtrace::capture()` 返回结构化的回溯对象，通过 `.frames()` 迭代获取每一帧的程序计数器（PC）地址。旧版直接返回 `[Option<u32>; 10]` 数组。

### 2.2 帧指针的支持

回溯能力依赖帧指针（frame pointer），在编译选项中启用：

```toml
# .cargo/config.toml
rustflags = ["-C", "force-frame-pointers"]
```

帧指针是函数调用栈的"链表"——每个栈帧开头保存了上一个栈帧的地址。通过沿着帧指针链回溯，就能得到完整的调用栈。Rust 默认会在优化时省略帧指针以节省一个寄存器，`force-frame-pointers` 强制保留。

### 2.3 地址解析

捕获到的 PC 地址如 `0x42001234` 是裸地址。要解析到函数名和源码位置，需要用到编译产物中的 ELF 文件：

```bash
# 使用 addr2line 解析地址
riscv32-esp-elf-addr2line -e target/riscv32imc-unknown-none-elf/release/epd-reader 0x42001234
```

这会输出类似 `src/wifi.rs:123` 的结果。虽然不能在设备上直接做这个解析，但通过串口日志记录地址，事后分析时可以精确定位崩溃位置。

## 3. 错误日志持久化

### 3.1 存储结构

```rust
// src/storage.rs:168-172
#[derive(Debug, Default, Clone)]
pub struct ErrorLogStorage {
    pub error_count: u32,
    pub last_error: heapless::String<200>,
}
```

错误日志存储在 Flash 的 NVS 分区，包含：
- `error_count`：累计 panic 次数
- `last_error`：最近一次 panic 的详细信息（最多 200 字符）

### 3.2 启动时检查

如果启用了 `enable_debug` feature，启动时会检查是否有错误日志：

```rust
// src/pages/main_page.rs:55-64
#[cfg(feature = "enable_debug")]
if let Ok(error_log) = crate::storage::ErrorLogStorage::read() {
    if error_log.error_count > 0 {
        println!("=== 检测到错误日志，进入调试模式 ===");
        println!("错误计数: {}", error_log.error_count);
        println!("最新错误信息:");
        println!("{}", error_log.last_error);
        page_index = 5;  // 直接进入 DebugPage
    }
}
```

这样开发者可以在重启后通过 DebugPage 查看上次崩溃的详细信息。

## 4. DebugPage

DebugPage 提供了一个简单的界面来查看错误日志，并支持清除操作：

- **Key1 长按**：清除错误日志（`error_count = 0`，`last_error = ""`）
- **Key3**：退出 DebugPage

它的 `render()` 方法直接将 `ErrorLogStorage` 的内容绘制到屏幕上。

## 5. 串口调试

### 5.1 esp-println

```rust
use esp_println::println;
println!("wifi connected: {:?}", state);
```

`esp-println` 通过 UART 输出调试信息。在 `cargo run` 时，`espflash monitor` 会自动打开串口监视器，所有 `println!` 输出都会显示在终端中。

### 5.2 log 框架

项目同时启用了 `log-04` feature，可以使用 `log` crate 的分级日志：

```rust
use log::trace;
trace!("test trace");  // 只有启用了 log 的项目才输出
```

`log` 框架支持 `error!`、`warn!`、`info!`、`debug!`、`trace!` 五个级别，可以在编译时过滤不需要的日志级别，减少串口输出量。

## 6. 常见错误模式

### 6.1 unwrap() panic

嵌入式代码中大量使用 `unwrap()`：

```rust
let mut epd = EpdControl::new(...).unwrap();
spawner.spawn(display::render(...)).unwrap();
```

如果 `EpdControl::new` 失败（如 SPI 通信错误），程序会 panic。Panic handler 会记录到 Flash，下次启动可以在 DebugPage 中看到。

### 6.2 栈溢出

ESP32-C3 的默认栈大小有限。大的结构体放在栈上可能导致栈溢出——`HolidayStorage` 的分块写入（第 6 篇）就是为了避免这个问题。

### 6.3 内存不足

80KB 的堆可能不够用，特别是 WiFi + TLS 同时需要缓冲区时。`esp-alloc` 会在内存不足时返回 `None`，调用 `unwrap()` 就会 panic。

## 7. 调试工具链

| 工具 | 用途 |
|------|------|
| `espflash monitor` | 串口监视器，查看 `println!` 输出 |
| `addr2line` | 将 PC 地址解析为源码位置 |
| `esp-backtrace` | RISC-V 调用栈回溯 |
| `enable_debug` feature | 启动时自动进入调试页面 |
| Flash 错误日志 | Panic 信息持久化，重启后可查看 |

## 小结

| 技术点 | 要点 |
|--------|------|
| Panic Handler | 自定义处理，捕获信息写入 Flash |
| Backtrace | `force-frame-pointers` + `esp-backtrace` |
| 错误日志 | Flash 持久化，重启后可在 DebugPage 查看 |
| 串口调试 | `esp-println` + `log` 框架 |
| 地址解析 | 事后用 `addr2line` 将 PC 地址映射到源码 |

嵌入式调试的核心挑战是**信息不足**——你无法attach 调试器，无法实时查看变量。本项目的策略是：**尽可能多地捕获和保存崩溃信息**，让事后分析成为可能。
