# 第 3 篇：电子墨水屏驱动与渲染架构

> 用 Rust 构建ESP32-C3 电子墨水屏阅读器 · 系列文章

## 引言

电子墨水屏（E-Paper Display，EPD）是一种非常特殊的显示技术。它不像 LCD 或 OLED 那样主动发光，而是通过电场控制微胶囊中带电颜料的移动来呈现图像。它的核心优势是：**断电后画面不消失**——这正是电子书阅读器选择它的原因。

但这个特性也带来了独特的工程挑战：刷新速度慢（几百毫秒到数秒）、有"鬼影"问题、刷新策略需要精心设计。本文将从驱动层到渲染架构，完整解析本项目如何驾驭电子墨水屏。

## 1. 电子墨水屏的工作原理

### 1.1 基本原理

电子墨水屏的每个像素包含一个透明微胶囊，里面有带正电的白色粒子和带负电的黑色粒子。施加正电压时，白色粒子移到顶部，像素显示白色；施加负电压则相反。

这意味着：
- **保持画面不需要耗电**——粒子停留在原位
- **改变画面才需要耗电**——需要施加电压移动粒子
- **刷新有延迟**——粒子物理移动需要时间

### 1.2 全刷与局部刷新

本项目使用的 Waveshare 电子墨水屏支持两种刷新模式：

| 模式 | 耗时 | 特点 |
|------|------|------|
| **Full（全刷）** | ~2-3 秒 | 黑白交替闪烁，完全清除残影 |
| **Quick（局部刷新）** | ~0.3 秒 | 不闪烁，但会积累鬼影 |

鬼影（Ghosting）是电子墨水屏特有的问题：上次显示的内容会隐约残留在新画面上。局部刷新虽然快，但每刷新几次就需要做一次全刷来清除鬼影。

## 2. 屏幕驱动层

### 2.1 条件编译选择屏幕

项目通过 feature flags 支持多种屏幕尺寸，在 `display.rs` 中通过类型别名统一抽象：

```rust
// src/display.rs:31-44
#[cfg(feature = "epd2in9")]
pub type EpdDisplay = Display2in9;          // 296×128 像素
#[cfg(feature = "epd2in9")]
pub type EpdControl<SPI, BUSY, DC, RST, DELAY> = Epd2in9<SPI, BUSY, DC, RST, DELAY>;

#[cfg(feature = "epd4in2")]
pub type EpdDisplay = Display4in2;          // 400×300 像素
#[cfg(feature = "epd4in2")]
pub type EpdControl<SPI, BUSY, DC, RST, DELAY> = Epd4in2<SPI, BUSY, DC, RST, DELAY>;

#[cfg(feature = "epd2in7")]
pub type EpdDisplay = Display2in7;          // 264×176 像素
```

上层代码统一使用 `EpdDisplay` 和 `EpdControl` 类型，不需要关心具体的屏幕型号。切换屏幕只需修改编译参数。

### 2.2 屏幕初始化

屏幕驱动通过 `epd-waveshare` crate（自定义 fork）实现。在渲染任务中完成初始化：

```rust
// src/display.rs:80-100
#[embassy_executor::task]
pub async fn render(
    mut spi_device: &'static mut ActualSpi<'static>,
    busy: esp_hal::peripherals::GPIO6<'static>,
    rst: esp_hal::peripherals::GPIO7<'static>,
    dc: esp_hal::peripherals::GPIO20<'static>,
) {
    // GPIO 配置
    let busy = Input::new(busy, esp_hal::gpio::InputConfig::default().with_pull(Pull::Down));
    let rst = Output::new(rst, Level::High, esp_hal::gpio::OutputConfig::default());
    let dc = Output::new(dc, Level::High, esp_hal::gpio::OutputConfig::default());

    // 初始化 EPD 控制器
    let mut epd = EpdControl::new(
        &mut spi_device, busy, dc, rst, &mut embassy_time::Delay
    ).unwrap();

    // 初始化帧缓冲区
    let mut display: EpdDisplay = EpdDisplay::default();
    display.clear_buffer(Color::White);

    // 将帧缓冲区存入全局静态变量
    unsafe {
        core::ptr::addr_of_mut!(DISPLAY).write(Some(display));
    }
    // ...
}
```

EPD 控制器需要 4 个 GPIO：
- **BUSY**：屏幕忙状态信号（屏幕在刷新时拉高，完成后拉低）
- **RST**：复位信号
- **DC**：数据/命令选择信号
- **SPI**：数据传输通过前面讲的共享 SPI 总线

初始化完成后，帧缓冲区被存入全局 `static mut DISPLAY`，供页面代码绘制使用。

### 2.3 SPI 设备类型

EPD 使用的 SPI 设备类型在 esp-hal v1.x 下是这样定义的：

```rust
// src/display.rs:70
type ActualSpi<'a> = CriticalSectionDevice<
    'a,
    Spi<'a, esp_hal::Blocking>,
    Output<'a>,
    embedded_hal_bus::spi::NoDelay,
>;
```

这个类型看起来复杂，但拆解后很清晰：
- `CriticalSectionDevice`：带临界区互斥的 SPI 设备（上一篇文章详解过）
- `Spi<'a, esp_hal::Blocking>`：阻塞模式的 SPI 总线
- `Output<'a>`：CS 片选引脚
- `NoDelay`：不需要 CS 到数据之间的延迟

## 3. 生产者-消费者渲染架构

### 3.1 架构设计

本项目最大的架构亮点之一是显示系统的**解耦设计**。页面代码不直接操作屏幕硬件，而是通过 Channel 发送渲染请求：

```
┌──────────────┐    RENDER_CHANNEL    ┌──────────────┐    SPI    ┌─────────┐
│  页面 (Pages) │ ──── RenderInfo ──► │ render task  │ ────────► │ EPD 屏幕 │
│  (生产者)      │                      │ (消费者)      │           │         │
└──────────────┘                       └──────────────┘           └─────────┘
```

Channel 的定义：

```rust
// src/display.rs:67-68
pub static RENDER_CHANNEL: Channel<CriticalSectionRawMutex, RenderInfo, 64> = Channel::new();
pub static QUICKLY_LUT_CHANNEL: Channel<CriticalSectionRawMutex, bool, 64> = Channel::new();
```

渲染请求携带的信息：

```rust
// src/display.rs:26-29
pub struct RenderInfo {
    pub time: i32,        // 预留时间戳
    pub need_sleep: bool, // 渲染后是否让屏幕进入睡眠
}
```

### 3.2 为什么这样设计？

**解耦页面和硬件**。页面代码只需要：绘制到帧缓冲区 → 发送渲染请求。它不需要知道 SPI 总线的存在，不需要知道屏幕是否在忙，也不需要等待屏幕刷新完成。

**避免阻塞**。电子墨水屏的刷新耗时几百毫秒到几秒。如果页面代码直接等待刷新完成，整个 UI 逻辑会被阻塞。通过 Channel，页面代码在发送请求后立即返回，可以继续处理下一个事件。

**串行化访问**。Channel 保证同一时间只有一个渲染请求被处理，避免了多个页面同时操作屏幕的冲突。

### 3.3 页面如何绘制？

页面代码的渲染流程如下：

```rust
// 典型的页面渲染流程（伪代码）
fn render_page() {
    let display = display::display_mut().unwrap();  // 获取帧缓冲区
    display.clear_buffer(Color::White);             // 清屏

    // 使用 embedded-graphics 绑定绘制
    Text::new("Hello", Point::new(10, 20), style).draw(display);

    // 发送渲染请求
    RENDER_CHANNEL.send(RenderInfo { time: 0, need_sleep: false }).await;
}
```

`display_mut()` 返回帧缓冲区的可变引用，页面可以自由绘制。绘制完成后通过 `RENDER_CHANNEL.send()` 通知渲染任务。

### 3.4 渲染任务：消费者的工作

渲染任务在独立 Embassy task 中运行，循环等待渲染请求：

```rust
// src/display.rs:111-198（核心逻辑简化）
loop {
    let render_sign = receiver.receive();           // 等待渲染请求
    let quickly_lut = quickly_lut_receiver.receive(); // 或刷新模式切换

    match select(render_sign, quickly_lut).await {
        Either::First(render_info) => {
            // 收到渲染请求
            if is_sleep {
                epd.wake_up(&mut spi_device, &mut embassy_time::Delay);
                is_sleep = false;
            }

            let buffer = unsafe {
                (*core::ptr::addr_of_mut!(DISPLAY)).as_mut().unwrap().buffer()
            };

            // 检查是否需要全刷（非 2.7 寸路径）
            let need_clear = take_need_clear();       // 仅冷启动首帧为 true
            let need_force_full = (get_render_times() % get_full_refresh_times() == 0 || need_clear)
                && refresh_lut == RefreshLut::Quick;
            if need_force_full {
                spi_device = set_refresh_mode(RefreshLut::Full, &mut epd, spi_device);
            }

            // 将帧缓冲区数据发送到屏幕
            epd.update_and_display_frame(&mut spi_device, buffer, &mut embassy_time::Delay)
                .expect("render failed");

            if need_force_full {
                spi_device = set_refresh_mode(RefreshLut::Quick, &mut epd, spi_device);
            }

            // 如果需要，让屏幕进入睡眠
            if render_info.need_sleep {
                is_sleep = true;
                epd.sleep(&mut spi_device, &mut embassy_time::Delay);
            }
        },
        Either::Second(v) => {
            // 切换刷新模式
            refresh_lut = if v { RefreshLut::Quick } else { RefreshLut::Full };
            spi_device = set_refresh_mode(refresh_lut, &mut epd, spi_device);
        },
    }
}
```

### 3.5 刷新策略

系统维护一个渲染计数器，每 N 次局部刷新后强制全刷一次以清残影。N 的**平台默认值**是编译期常量（2.7 寸屏 20 次，其余 5 次），但可以在运行时被设置页 / Web 配置覆盖：

```rust
// src/display.rs
#[cfg(feature = "epd2in7")]
pub const DEFAULT_FULL_REFRESH_TIMES: u32 = 20;
#[cfg(not(feature = "epd2in7"))]
pub const DEFAULT_FULL_REFRESH_TIMES: u32 = 5;

// 运行时缓存（0 表示尚未加载，回退到平台默认）
static mut FULL_REFRESH_TIMES: u32 = 0;
```

`FULL_REFRESH_TIMES` 从 Flash 的 `DisplayStorage.full_refresh_times` 加载，并由 `reload_full_refresh_times()` 钳制到 `1..=100`——这步钳制是必须的，旧固件下该分区可能是 Flash 原始字节（0 或超大值），而 `get_render_times() % N` 在 N=0 时会触发整除零 panic。

```rust
pub fn reload_full_refresh_times() {
    let raw = crate::storage::DisplayStorage::read().map(|d| d.full_refresh_times).unwrap_or(0);
    let v = if (1..=100).contains(&raw) { raw } else { DEFAULT_FULL_REFRESH_TIMES };
    unsafe { core::ptr::addr_of_mut!(FULL_REFRESH_TIMES).write(v); }
}
```

渲染计数器本身存储在 RTC 内存中，深度睡眠唤醒后仍然保持：

```rust
// src/display.rs
#[ram(unstable(rtc_fast))]
static mut RENDER_TIMES: u32 = 0;
```

此外还有一个冷启动清屏标记 `NEED_CLEAR`（rtc_fast 初值为 true，首次渲染后复位为 false）。深睡唤醒不会重置它，所以只有真正冷启动（上电）的 第一帧 才会强制一次全刷清残影：

```rust
#[ram(unstable(rtc_fast))]
static mut NEED_CLEAR: bool = true;
```

### 3.6 2.7 寸屏的特殊处理

2.7 寸屏幕的局部刷新机制与众不同——它需要维护一个"旧帧"引用来计算差分。但在深度睡眠中，屏幕控制器的 RAM 会丢失，唤醒后第一帧必须重新建立基准。

```rust
// src/display.rs:53-58
#[cfg(feature = "epd2in7")]
static mut PREV_BUFFER: [u8; 5808] = [0xFF; 5808];  // 上一帧的备份
#[cfg(feature = "epd2in7")]
static mut PREV_BUFFER_VALID: bool = false;           // 是否有有效的旧帧
```

唤醒后的第一帧渲染逻辑：

```rust
// src/display.rs:129-163（简化）
if need_base_map {
    if prev_valid {
        // 有旧帧：使用静默过渡（不闪烁）
        epd.set_base_map_quiet(&mut spi_device, prev, buffer, &mut embassy_time::Delay);
    } else {
        // 没有旧帧：必须全刷
        epd.set_base_map(&mut spi_device, buffer, &mut embassy_time::Delay);
    }
    need_base_map = false;
} else {
    // 正常局部刷新
    epd.update_and_display_frame_partial(&mut spi_device, buffer, &mut embassy_time::Delay);
}
// 每次渲染后保存当前帧
unsafe {
    (*core::ptr::addr_of_mut!(PREV_BUFFER)).copy_from_slice(buffer);
    core::ptr::addr_of_mut!(PREV_BUFFER_VALID).write(true);
}
```

这个设计非常精妙：通过在 RAM 中保存上一帧的备份，唤醒后可以做一次"静默"的基准帧设置，避免全刷时的闪烁。

## 4. 帧缓冲区与 embedded-graphics

### 4.1 帧缓冲区

电子墨水屏的帧缓冲区是一个字节数组，每个 bit 代表一个像素（0 = 白，1 = 黑）。以 2.9 寸屏为例：

```
296 × 128 像素
= 296 × 128 / 8 字节
= 4736 字节 (约 4.6KB)
```

4.2 寸屏的帧缓冲区更大：400 × 300 / 8 = 15000 字节（约 14.6KB）。在 400KB SRAM 的 ESP32-C3 上，这是一笔不小的内存开销。

### 4.2 embedded-graphics 绑定绘制

本项目使用 `embedded-graphics` crate 作为图形绘制框架。它的核心是 `DrawTarget` trait：

```rust
// embedded-graphics 定义的 trait
pub trait DrawTarget {
    type Color;
    fn draw_iter<I>(&mut self, pixels: I) -> Result<(), Self::Error>
    where I: IntoIterator<Item = Pixel<Self::Color>>;
}
```

`EpdDisplay` 实现了 `DrawTarget`，所以可以使用所有 `embedded-graphics` 的绑定绘制功能：

```rust
// 文字绘制
let style = MonoTextStyleBuilder::new()
    .font(&FONT_6X10)
    .text_color(White)
    .background_color(Black)
    .build();
Text::new("Hello", Point::new(10, 20), style).draw(display);

// 矩形绘制
Rectangle::new(Point::new(0, 0), Size::new(100, 50))
    .into_styled(PrimitiveStyle::with_fill(Black))
    .draw(display);

// 自定义 Widget 绘制
calendar.draw(display);  // 任何实现了 Drawable trait 的组件
```

### 4.3 中文渲染

中文显示是本项目的重要需求。通过 `u8g2-fonts` crate 加载 GB2312 编码的中文字体：


`u8g2_font_wqy15_t_gb2312` 是文泉驿字体，支持完整的 GB2312 中文字符集。15 表示字高 15 像素——在 296×128 的小屏幕上，每行约显示 20 个汉字，7 行文字。

使用方式：


`render_aligned` 支持水平和垂直方向的多种对齐方式，非常适合 UI 布局。

## 5. 待机壁纸系统

待机壁纸是电子墨水屏的一个巧妙应用——因为断电后画面不消失，待机画面可以作为设备的"外观"。

### 5.1 存储位置

壁纸存储在 Flash 的 `storage` 分区（`0x310000` 起始，128KB）：

```rust
// src/flash_sleep.rs:9
const STORAGE_OFFSET: u32 = 0x310000;
```

存储格式：8 字节头 + 像素数据。头部包含魔数（`0x534C4550`，即 "SLEP"）和像素数据长度。

### 5.2 BMP 转换与缩放

用户上传的 BMP 图片需要转换为屏幕可用的 1-bit 格式，并缩放到目标尺寸：

```rust
// src/flash_sleep.rs:38-109（核心逻辑简化）
pub fn bmp_to_pixels(raw_bmp: &[u8], target_w: u32, target_h: u32) -> Option<Vec<u8>> {
    // 解析 BMP 头
    let bpp = ...; // 支持 1-bit, 24-bit, 32-bit

    // 逐像素缩放
    for dy in 0..target_h {
        let src_y = dy * bmp_h / target_h;  // 最近邻缩放
        for dx in 0..target_w {
            let sx = dx * bmp_w / target_w;
            let is_black = match bpp {
                1 => /* 读取 1-bit 像素 */,
                24 | 32 => {
                    // RGB -> 灰度 -> 二值化
                    let (b, g, r) = ...;
                    (r * 299 + g * 587 + b * 114) / 1000 < 128  // ITU-R BT.601
                },
            };
            // 打包为 1-bit
            if is_black {
                pixels[row_offset + dx / 8] |= 1 << (7 - (dx % 8));
            }
        }
    }
}
```

灰度到二值的转换使用 ITU-R BT.601 标准的亮度公式：`Y = 0.299R + 0.587G + 0.114B`，阈值为 128。

### 5.3 流式读取渲染

壁纸图像可能很大（300×400/8 = 15000 字节），为了避免一次性分配大量内存，渲染时逐行从 Flash 读取：

```rust
// src/flash_sleep.rs:219-246
pub fn draw_sleep_image<D>(display: &mut D) -> bool {
    // 逐行读取
    let mut row_buf = [0u8; ROW_BYTES];
    for y in 0..SLEEP_IMG_H {
        let offset = STORAGE_OFFSET + HEADER_SIZE as u32 + y as u32 * ROW_BYTES as u32;
        read_flash(offset, &mut row_buf);
        // 逐像素绘制到帧缓冲区
        for x in 0..SLEEP_IMG_W {
            if (row_buf[x / 8] >> (7 - x % 8)) & 1 != 0 {
                Pixel(Point::new(x as i32, y as i32), BinaryColor::On).draw(display);
            }
        }
    }
}
```

每次只读一行（约 38 字节），内存占用极低。

### 5.4 崩溃安全写入

壁纸写入采用"先清头部、写数据、再写有效头部"的三步策略：

```rust
// src/flash_sleep.rs:195-217
pub fn save_sleep_image(raw_bmp: &[u8]) -> Result<(), &'static str> {
    let pixels = bmp_to_pixels(raw_bmp, SLEEP_IMG_W, SLEEP_IMG_H)?;

    // 步骤 1：清除头部（标记为无效）
    write_flash(STORAGE_OFFSET, &[0u8; 8])?;

    // 步骤 2：写入像素数据
    write_flash(STORAGE_OFFSET + HEADER_SIZE as u32, &pixels)?;

    // 步骤 3：写入有效头部（标记为有效）
    header[0..4].copy_from_slice(&SLEEP_IMAGE_MAGIC.to_le_bytes());
    header[4..8].copy_from_slice(&(total_size as u32).to_le_bytes());
    write_flash(STORAGE_OFFSET, &header)?;

    Ok(())
}
```

如果在写入像素数据过程中断电，重启后头部是无效的（全零），系统会使用默认壁纸。只有完整写入后才将头部标记为有效。这是一种简单但有效的崩溃安全策略。

## 6. 渲染流水线总结

完整的渲染流水线如下：

```
用户操作 / 定时器
      │
页面事件回调
      │
修改页面状态
      │
display_mut() → 获取帧缓冲区（static mut）
      │
embedded-graphics API 绑定绘制
(Text / Rectangle / Widget / Font)
      │
RENDER_CHANNEL.send(RenderInfo)
      │（异步 Channel，不阻塞）
      │
render task 接收请求
      │
检查屏幕是否在睡眠 → wake_up()
      │
检查刷新计数 → 是否需要全刷
      │
epd.update_and_display_frame(buffer)
      │（通过 SPI 发送到屏幕，等待 BUSY 信号）
      │
检查 need_sleep → 是否让屏幕睡眠
      │
Timer::after(50ms) → 等待稳定，继续循环
```

| 帧缓冲区 | 1-bit 二值图，通过 core::ptr 全局访问 |
| 图形绘制 | embedded-graphics DrawTarget + u8g2 中文渲染 |
| 待机壁纸 | Flash 存储 + BMP 转换 + 流式渲染 + 崩溃安全 |

核心设计思想：**将耗时的硬件操作封装在独立任务中，通过异步 Channel 解耦，让页面代码专注于业务逻辑而非硬件细节**。

在下一篇文章中，我们将深入 Embassy 异步运行时和事件驱动架构，看看按键事件是如何从 GPIO 电平变化变成页面响应的。
