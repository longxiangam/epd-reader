# 第 6 篇：数据持久化与文件系统

> 用 Rust 构建ESP32-C3 电子墨水屏阅读器 · 系列文章

## 引言

嵌入式设备需要在断电后保存数据——WiFi 配置、天气缓存、阅读进度、错误日志……在 PC 上你可能会用 SQLite 或文件系统，但在只有 4MB Flash 的 ESP32-C3 上，每一字节都要精打细算。

本项目使用两套存储方案：ESP32 内部 Flash 的 NVS 分区用于配置数据，MicroSD 卡通过 SPI 用于大容量存储（电子书、图片）。本文将深入它们的实现细节。

## 1. Flash 存储：零拷贝序列化

### 1.1 Flash 访问接口

ESP32 的 Flash 通过 `esp-storage` crate 访问，它实现了 `embedded_storage` 的 `ReadStorage` 和 `Storage` trait：

```rust
// src/storage.rs:11-26
pub fn write_flash(flash_addr: u32, bytes: &[u8]) -> Result<(), FlashStorageError> {
    let flash = unsafe { esp_hal::peripherals::FLASH::steal() };
    let mut flash = FlashStorage::new(flash);
    flash.write(flash_addr, bytes)
}

pub fn read_flash(flash_addr: u32, bytes: &mut [u8]) -> Result<(), FlashStorageError> {
    let flash = unsafe { esp_hal::peripherals::FLASH::steal() };
    let mut flash = FlashStorage::new(flash);
    flash.read(flash_addr, bytes)
}
```

`FLASH::steal()` 是 esp-hal v1.x 新增的 API——它绕过所有权系统直接获取 Flash 外设的引用。之所以可以这样做，是因为 Flash 操作不需要像 SPI 那样的独占访问（它使用内部总线），而且整个程序只有一处操作 Flash。

### 1.2 零拷贝序列化

本项目使用一种极其简洁的序列化方案——直接将结构体的内存表示写入 Flash：

```rust
// src/storage.rs:43-49
fn serialize_storage<T>(storage: &T) -> [u8; size_of::<T>()] {
    unsafe { ptr::read(storage as *const _ as *const [u8; size_of::<T>()]) }
}

fn deserialize_storage<T>(data: &[u8]) -> T {
    unsafe { ptr::read(data.as_ptr() as *const T) }
}
```

`serialize_storage` 将任意结构体的内存布局重新解释为字节数组，`deserialize_storage` 则相反。这本质上是 C 语言中常见的 `memcpy` 序列化，但用 Rust 的 `ptr::read` 表达。

**优点**：零堆分配、零拷贝、极快的序列化速度。
**限制**：结构体的内存布局必须与 Flash 中的一致。如果修改了结构体定义（增减字段、改变类型），Flash 中的旧数据就无法正确反序列化。

### 1.3 存储布局

NVS 分区（0x9000 起始）被划分为连续的存储块，每个块存放一种数据：

```rust
// src/storage.rs:104-175
const NVS_OFFSET: usize = 0x9000;

const VERSION_STORAGE_OFFSET: usize = NVS_OFFSET + 0x00;
// VersionStorage { version: u32, init_tag: u32 }

const WIFI_STORAGE_OFFSET: usize = VERSION_STORAGE_OFFSET + size_of::<VersionStorage>();
// WifiStorage { ssid: String<32>, password: String<64>, wifi_finish: bool }

const WEATHER_STORAGE_OFFSET: usize = WIFI_STORAGE_OFFSET + size_of::<WifiStorage>();
// WeatherStorage { token: String<64>, city: String<32>, sync_time: u64, data: Option<DailyResult> }

const SLEEP_STORAGE_OFFSET: usize = WEATHER_STORAGE_OFFSET + size_of::<WeatherStorage>();
// SleepStorage { read_sleep_seconds: u64, weather_sleep_seconds: u64 }

const HOLIDAY_OFFSET: usize = ...;
// HolidayStorage { token, sync_time, holiday_response }

const ERROR_LOG_OFFSET: usize = ...;
// ErrorLogStorage { error_count: u32, last_error: String<200> }
```

每个存储块的偏移量是前一个块的偏移量加上其大小——一种简单但有效的顺序分配。

### 1.4 impl_storage! 宏

为了避免为每种存储类型重复实现读写方法，项目定义了一个宏：

```rust
// src/storage.rs:83-103
macro_rules! impl_storage {
    ($type:ty, $offset:expr) => {
        impl NvsStorage for $type {
            fn read() -> Result<Self, FlashStorageError> {
                let mut buffer = [0u8; size_of::<Self>()];
                read_flash($offset as u32, &mut buffer)?;
                Ok(deserialize_storage(&buffer))
            }
            fn write(&self) -> Result<(), FlashStorageError> {
                let data = serialize_storage(self);
                write_flash($offset as u32, &data)
            }
        }
    };
}

// 一行代码为类型实现读写
impl_storage!(VersionStorage, VERSION_STORAGE_OFFSET);
impl_storage!(WifiStorage, WIFI_STORAGE_OFFSET);
impl_storage!(WeatherStorage, WEATHER_STORAGE_OFFSET);
impl_storage!(SleepStorage, SLEEP_STORAGE_OFFSET);
```

`NvsStorage` trait 定义了统一的读写接口：

```rust
pub trait NvsStorage {
    fn read() -> Result<Self, FlashStorageError> where Self: Sized;
    fn write(&self) -> Result<(), FlashStorageError>;
}
```

### 1.5 分块写入：解决栈溢出

`HolidayStorage` 包含一个较大的 `HolidayResponse` 结构体，整体序列化可能超过栈空间。项目为此实现了分块写入：

```rust
// src/storage.rs:52-74
fn write_storage_chunked<T>(storage: &T, offset: u32) -> Result<(), FlashStorageError> {
    const CHUNK_SIZE: usize = 128;  // 每次只处理 128 字节
    let total_size = size_of::<T>();
    let storage_ptr = storage as *const T as *const u8;

    for chunk_offset in (0..total_size).step_by(CHUNK_SIZE) {
        let chunk_size = core::cmp::min(CHUNK_SIZE, total_size - chunk_offset);
        let mut chunk = [0u8; CHUNK_SIZE];  // 固定大小的栈缓冲区

        unsafe {
            ptr::copy_nonoverlapping(
                storage_ptr.add(chunk_offset),
                chunk.as_mut_ptr(),
                chunk_size
            );
        }
        write_flash(offset + chunk_offset as u32, &chunk[..chunk_size])?;
    }
    Ok(())
}
```

关键点：无论结构体多大，栈上始终只分配 128 字节的缓冲区。通过循环逐块写入，避免了 `serialize_storage` 中 `[u8; size_of::<T>()]` 在栈上分配大数组的风险。

### 1.6 版本检查与初始化

启动时检查 Flash 中的初始化标记，如果不存在或不匹配则重新初始化所有存储区域：

```rust
// src/storage.rs:210-225
pub async fn enter_process() {
    let version_storage = VersionStorage::read();
    match version_storage {
        Ok(v) => {
            if v.init_tag != INIT_TAG {  // 0x1234abcb
                init_storage_area();     // 结构体变更后需要更新 INIT_TAG
            }
            let wifi = WifiStorage::read().unwrap();
            WIFI_INFO.lock().await.replace(wifi);
        }
        Err(_) => {
            init_storage_area();  // 首次启动
        }
    }
}
```

`INIT_TAG` 是一个魔数常量。每次修改存储结构体后需要更改这个值，这样下次启动时会检测到不匹配并重新初始化。

## 2. SD 卡文件系统

### 2.1 挂载与类型抽象

SD 卡通过 SPI 总线使用 `embedded-sdmmc` crate 操作。为了简化泛型，项目预定义了类型别名：

```rust
// src/sd_mount.rs:30-36
pub type SdCsPin = Output<'static>;
pub type ActualSpi = CriticalSectionDevice<'static, Spi<'static, esp_hal::Blocking>, SdCsPin, ...>;
pub type ActualSdCard = SdCard<&'static mut ActualSpi, Delay>;
pub type ActualVolumeManager = VolumeManager<ActualSdCard, TimeSource>;
pub type ActualDirectory<'a> = Directory<'a, ActualSdCard, TimeSource, 4, 4, 1>;
pub type ActualFile<'a> = File<'a, ActualSdCard, TimeSource, 4, 4, 1>;
```

泛型参数 `4, 4, 1` 分别是最大卷数、最大打开目录数、最大打开文件数。在资源受限环境中，这些参数需要精心控制。

SD 卡在 `main.rs` 中挂载后存入全局 Mutex：

```rust
// src/main.rs:182-185
let sdcard = SdCard::new_with_options(spi_bus_sd, embassy_time::Delay, AcquireOpts {
    use_crc: false,           // 关闭 CRC 校验（某些卡不兼容）
    acquire_retries: 50,      // 初始化重试 50 次
});
let volume_mgr = VolumeManager::new(sdcard, TimeSource);
let sd_mount = SdMount::new(volume_mgr);
SD_MOUNT.lock().await.replace(sd_mount);
```

### 2.2 长文件名（LFN）支持

`embedded-sdmmc` 的 LFN 支持有限，项目手动实现了 LFN 目录项的写入。这是一个复杂但必要的工作——中文书名需要长文件名支持。

实现分为三个阶段：

**阶段 1**：生成唯一的 8.3 短文件名

```rust
// src/sd_mount.rs:473-495

    let mut max_num: u32 = 0;
    // 扫描目录，找到最大的 BKxxxx 编号
    books_dir.iterate_dir_lfn(&mut lfn_buffer, |dir_entry, _lfn| {
        if name_str.starts_with("BK") && name_str.contains('~') {
            // 提取编号
            max_num = ...;
        }
    });
    // 生成如 BK0001~1.TXT 的短名
    let short = format!("BK{:04X}~1.TXT", max_num + 1);
}
```

**阶段 2**：用短名创建文件（让 FAT 分配簇和目录项）

**阶段 3**：在目录项前插入 LFN 条目

```rust
// src/sd_mount.rs:552-588（简化）
// 读取 FAT 目录扇区
BlockDevice::read(block_dev, &mut blocks, entry_block);
let data = &mut blocks[0].contents;

// 将短名目录项后移，腾出空间
let shift_bytes = num_lfn_entries * 32;
for i in (src_start..512 - shift_bytes).rev() {
    data[i + shift_bytes] = data[i];
}

// 写入 LFN 条目
for seq in 0..num_lfn_entries {
    let lfn_entry = build_lfn_entry(seq_byte, checksum, chunk);
    data[offset..offset + 32].copy_from_slice(&lfn_entry);
}

// 写回扇区
BlockDevice::write(block_dev, &blocks, entry_block);
```

这涉及 FAT32 文件系统的底层操作——直接读写目录扇区、计算 LFN 校验和、构建 UTF-16LE 编码的目录项。工作量不小，但让设备能正确显示中文文件名。

## 3. 文本分页引擎

### 3.1 问题背景

电子书阅读的核心需求是将 TXT 文件按屏幕尺寸分页。但 TXT 文件没有"页"的概念——你需要根据字体宽度、屏幕尺寸来计算每页能显示多少文字。

难点在于：
- ASCII 字符宽度不固定（`'i'` 窄，`'W'` 宽）
- 中文字符固定 16 像素宽
- UTF-8 编码下，中文 3 字节、ASCII 1 字节
- 需要处理跨缓冲区的字符边界

### 3.2 字符类型检测

```rust
// src/epd2in9_txt.rs:533-559
fn char_type_width(byte: u8) -> (CharType, u8) {
    let temp = byte & 0b1111_0000;
    if temp == 0b1111_0000 { (Other, 4) }      // 4 字节 UTF-8
    else if temp == 0b1110_0000 { (Zh, 3) }     // 3 字节：中文
    else {
        let temp = byte & 0b1100_0000;
        if temp == 0b0000_0000 { (Ascii, 1) }   // ASCII
        else if temp == 0b1000_0000 { (Tail, 1) } // UTF-8 续字节
        else { (Other, 2) }                      // 2 字节 UTF-8
    }
}
```

通过 UTF-8 编码的首字节特征判断字符类型和字节长度。

### 3.3 分页算法

分页过程扫描整个文件，逐字符计算行宽和行数，当行数达到屏幕容量时记录一个分页点：

```rust
// src/epd2in9_txt.rs:90-207（简化）
while end_position < file_length {
    let mut line_width = 0;
    let mut lines_num = 0;

    while !my_file.is_eof() {
        let mut buffer = [0u8; 200];
        let num_read = my_file.read(&mut buffer).unwrap();

        let mut i = 0;
        while i < num_read {
            let byte = buffer[i];
            let (char_type, byte_num) = char_type_width(byte);

            match char_type {
                Ascii => {
                    if char == '\n' || char == '\r' {
                        lines_num += 1; line_width = 0;
                    } else {
                        line_width += ascii_width(char);
                    }
                }
                Zh | Other => { line_width += ZH_WIDTH; }  // 中文 16px
                Tail => {}  // 跳过续字节
            }

            end_position += byte_num as u32;
            i += byte_num;

            // 行宽超限，换行
            if line_width > display_width { lines_num += 1; line_width = 0; }

            // 行数达上限，换页
            if lines_num == display_lines {
                all_page_position_vec.push(end_position);
                // 写入索引文件...
                break;
            }
        }
    }
}
```

每次分页会将分页位置写入 `.idx` 索引文件。索引文件就是一个 `u32` 数组，每个值表示一页在 TXT 文件中的字节偏移量。

### 3.4 分页索引

大文件（> 1MB）不可能一次性加载所有分页信息。`BookPages` 结构体实现了按需加载：

```rust
// src/epd2in9_txt.rs:376-391
pub struct BookPages {
    pub total_page: u32,
    total_vec_nums: u32,           // 索引分片数量
    current_vec_num: u32,          // 当前加载的分片
    page_vec: Vec<u32, 1000>,      // 当前分片的分页位置
    need_read_page_vec: bool,      // 是否需要重新加载
}
```

每次最多加载 1000 个分页位置（4KB），跳页时按需加载对应分片。这样无论文件多大，内存占用都是恒定的。

### 3.5 ASCII 宽度表

由于 `u8g2` 字体的 ASCII 字符宽度各不相同，项目通过预生成的方式创建了宽度查找表：

```rust
// src/epd2in9_txt.rs:571-581
fn ascii_width(ch: char) -> u32 {
    let width = if ch == '"' { 3 }
        else if ch == '#' { 8 }
        else if ch == 'A' { 9 }
        // ... 所有 ASCII 可打印字符 ...
        else { 0 };
    if width > 0 { width + 2 } else { 0 }  // +2 是字符间距
}
```

虽然看起来笨拙，但在 `#![no_std]` 环境中没有更优雅的方式来存储这个查找表——它保证了精确的排版计算。

## 小结

| 技术点 | 要点 |
|--------|------|
| Flash 访问 | `esp-storage` + `FLASH::steal()` |
| 零拷贝序列化 | `ptr::read` 直接将结构体内存写入 Flash |
| 存储布局 | NVS 分区顺序分配，偏移量由 `size_of` 计算 |
| 分块写入 | 128 字节分块避免栈溢出 |
| impl_storage! 宏 | 一行代码为类型实现读写 trait |
| SD 卡 | `embedded-sdmmc` + SPI，类型别名简化泛型 |
| LFN 支持 | 手动实现 FAT32 LFN 目录项写入 |
| 文本分页 | 逐字符计算行宽，索引文件按需加载 |
| ASCII 宽度 | 预生成的逐字符宽度查找 |

核心思想：**在资源受限环境中，用最简单直接的方式解决持久化问题**。零拷贝序列化不是最优雅的方案，但它是最适合嵌入式环境的——快、小、省内存。
