# 第 1 篇：项目全景 — 从零搭建 ESP32-C3 嵌入式 Rust 开发环境

> 用 Rust 构建ESP32-C3 电子墨水屏阅读器 · 系列文章

## 引言

这是一个运行在 ESP32-C3 上的电子墨水屏阅读器，用 Rust 的 `#![no_std]` 写成——不链接标准库、没有操作系统，程序直接跑在裸机上。设备带 WiFi，能做这些事：读 TXT 电子书、看天气与温度趋势、看农历日历和节假日（含调休）、查沪深股票的分时/K 线/盘口、浏览 BMP 图片，外加手机扫码配网、IP 自动定位、定时刷新和深度睡眠省电。

本文是项目文档的第一篇，先讲整体结构和开发环境怎么搭起来。后面几篇按子系统拆开：SPI 总线共享、电子墨水屏驱动与渲染、Embassy 异步运行时与事件、WiFi 与 HTTP、Flash 持久化与 SD 卡、低功耗、UI 组件与页面、错误处理。

## 1. 从 `#![no_std]` 说起

在 PC 上写 Rust，你理所当然地使用 `String`、`Vec`、`println!`，这些都来自 Rust 标准库（`std`）。但在微控制器上，没有操作系统，没有文件系统，没有标准输入输出——标准库里的大多数功能都不可用。

所以嵌入式 Rust 程序的第一行，永远是：

```rust
#![no_std]
```

这行声明告诉编译器：不要链接标准库，我们只使用 `core`（Rust 的最小核心库，不依赖操作系统）。

接着，嵌入式程序也没有常规的 `main()` 入口——因为 C 运行时（crt0）不会帮我们调用它。所以还需要：

```rust
#![no_main]
```

那么谁来充当程序入口呢？在 ESP32 上，这由 `esp-rtos` 提供：

```rust
// src/main.rs
#![no_std]
#![no_main]

esp_bootloader_esp_idf::esp_app_desc!();  // 生成应用描述信息

#[esp_rtos::main]
async fn main(spawner: Spawner) -> ! {
    // 程序从此处开始执行
    // 注意返回类型是 -> !，即永不返回
}
```

`#[esp_rtos::main]` 是一个过程宏（proc macro），它会将 `main` 函数包装成 ESP32 的正确入口点，并设置好 Embassy 异步执行器。`-> !` 表示这个函数永远不会返回——在嵌入式世界里，程序就是一个无限循环。

### 不稳定的 nightly 特性

由于嵌入式 Rust 生态（esp-hal、Embassy 等）依赖一些尚未稳定的 Rust 特性，我们需要使用 nightly 版本的编译器，并在文件头部声明这些 feature：

```rust
// src/main.rs
#![feature(impl_trait_in_assoc_type)]   // trait 关联类型中使用 impl Trait
#![feature(type_alias_impl_trait)]      // 类型别名中使用 impl Trait
#![feature(generic_const_exprs)]        // 泛型常量表达式
```

这些特性在 embassy-net、embedded-graphics 等库中被广泛使用，是当前嵌入式 Rust 开发的"标配"。

## 2. ESP32-C3 与 RISC-V

ESP32-C3 是乐鑫（Espressif）推出的一款低成本、低功耗的微控制器，采用 **RISC-V 32 位指令集**（具体为 `riscv32imc`），主频 160MHz，内置 400KB SRAM 和 4MB Flash。

选它的原因很直接：
- **价格便宜**，模块单价十几元
- **有 WiFi**，ESP32 系列的标配
- **RISC-V 架构**，没有 ARM 的授权成本，也更适合学习
- **Rust 生态完善**，`esp-rs` 对 ESP32-C3 的支持是一等公民

关键的一点：ESP32-C3 **没有 MMU**（内存管理单元），这意味着它不能运行 Linux 等操作系统。我们的程序直接运行在"裸机"上，这也是 `#![no_std]` 的根本原因。

### 目标三元组

在 Rust 的交叉编译世界里，"目标三元组"（target triple）标识了编译的目标平台。ESP32-C3 对应的是：

```
riscv32imc-unknown-none-elf
```

拆解一下：
- `riscv32imc`：RISC-V 32 位，支持整数(I)、乘法(M)、压缩(C)扩展
- `unknown`：未知厂商
- `none`：没有操作系统
- `elf`：输出 ELF 格式

这个目标三元组贯穿整个项目配置。

## 3. 工具链配置

### 3.1 rust-toolchain.toml

Rust 项目通过 `rust-toolchain.toml` 文件声明所需的工具链：

```toml
# rust-toolchain.toml
[toolchain]
targets = ["riscv32imc-unknown-none-elf"]
```

这里只指定了 target，没有锁定具体的 nightly 版本。在实际项目中，你也可以锁定到一个特定日期的 nightly（如 `channel = "nightly-2024-09-19"`），但 esp-hal v1.x 已经对较新的 nightly 有良好的兼容性，一般不需要锁定。

### 3.2 安装工具链

首次开发前，需要安装目标平台和烧录工具：

```bash
# 添加 RISC-V 目标
rustup target add riscv32imc-unknown-none-elf

# 安装 espflash（用于编译、烧录、监控）
cargo install espflash

# （可选）安装 probe-rs，支持调试器烧录
cargo install probe-rs
```

### 3.3 .cargo/config.toml

Cargo 的构建配置文件定义了如何编译和运行：

```toml
# .cargo/config.toml
[target.riscv32imc-unknown-none-elf]
runner = "espflash flash --chip esp32c3 --baud 115200 --monitor --partition-table ./partitions.csv"

[build]
rustflags = [
  "-C", "link-arg=-Tlinkall.x",        # 内存布局链接脚本
  "-C", "force-frame-pointers",          # 保留帧指针，用于 backtrace 调试
]
target = "riscv32imc-unknown-none-elf"

[env]
CC = "riscv32-esp-elf-gcc"               # C 编译器（链接脚本需要）
AR = "riscv32-esp-elf-ar"
```

几个关键点：
- **runner**：指定 `cargo run` 时执行的命令。`espflash flash` 会编译、烧录并打开串口监控。`--partition-table ./partitions.csv` 告诉烧录器使用我们自定义的分区表
- **linkall.x**：ESP32 的内存布局链接脚本，定义了 RAM/Flash 的地址范围
- **force-frame-pointers**：强制保留栈帧指针，让 `esp-backtrace` 在程序崩溃时能打印调用栈

有了这个配置，只需一条命令就能编译并烧录到设备：

```bash
cargo run --release --features epd4in2
```

> 必须用 `--release`：固件体积已超过 debug 默认配置，`[profile.release] opt-level = "s"` 才能链接进 3MB 的 factory 分区。

## 4. 依赖生态全景

ESP32 嵌入式 Rust 的依赖生态是一个分层结构。先看本项目的 `Cargo.toml`：

```toml
# Cargo.toml（有删减）
[package]
name = "epd-reader"
edition = "2024"

[dependencies]
# ── ESP 官方生态 ──
esp-hal = { version = "1.1.0", features = ["esp32c3", "unstable", "log-04"] }
esp-radio = { version = "0.18.0", features = ["esp32c3", "wifi", "log-04", "unstable"] }
esp-rtos = { version = "0.3.0", features = ["esp32c3", "embassy", "esp-radio", "log-04"] }
esp-alloc = { version = "0.10.0", features = ["esp32c3", "global-allocator"] }
esp-storage = { version = "0.9.0", features = ["esp32c3"] }
esp-bootloader-esp-idf = { version = "0.5.0", features = ["esp32c3"] }
esp-backtrace = { version = "0.19.0", features = ["esp32c3", "println"] }
esp-println = { version = "0.17.0", features = ["esp32c3", "log-04"] }

# ── Embassy 异步框架 ──
embassy-executor = { version = "0.10.0", features = ["nightly"] }
embassy-time = "0.5.0"
embassy-net = { version = "0.9.0", features = ["dhcpv4", "medium-ethernet", "tcp", "dns", "udp"] }
embassy-sync = "0.8"

# ── 图形 ──
embedded-graphics = "0.8"
u8g2-fonts = { version = "0.7.1", features = ["embedded_graphics_textstyle"] }

# ── 网络 ──
reqwless = { version = "0.14.0", default-features = false }
embedded-tls = { version = "0.18", default-features = false }
no-std-net = "0.6"
sntpc = { version = "0.3", features = ["async"] }

# ── 存储 ──
embedded-sdmmc = "0.9.0"
```

这些依赖可以分成以下几个层次：

```
┌─────────────────────────────────────┐
│         应用层 (Pages/Widgets)       │  ← 我们的代码
├─────────────────────────────────────┤
│       服务层 (display, wifi, etc.)   │  ← 我们的代码
├─────────────────────────────────────┤
│     Embassy 异步框架                 │  ← embassy-executor/net/sync
├─────────────────────────────────────┤
│     ESP HAL + 驱动                   │  ← esp-hal / esp-radio / esp-alloc
├─────────────────────────────────────┤
│     硬件 (ESP32-C3)                  │
└─────────────────────────────────────┘
```

每个 ESP 相关的 crate 都需要指定 `features = ["esp32c3"]`，因为 ESP32 系列有多个型号（C3、C6、S2、S3 等），它们的寄存器定义和外设列表各不相同。

### feature flags 的作用

`esp-hal` 中的 `unstable` feature 值得特别说明。esp-hal v1.x 将一些 API 标记为"不稳定"，需要显式启用：

```toml
esp-hal = { version = "1.1.0", features = ["esp32c3", "unstable"] }
```

这些 API 可能会在未来版本中变更，但在当前版本中是功能完整的。对于实际项目，启用 `unstable` 是必要的——很多关键功能（如 RTC 内存属性 `#[ram(unstable(rtc_fast))]`）都在 unstable 特性之后。

## 5. Flash 分区表

ESP32 的 4MB Flash 不是一整块随便用的，它被划分成多个分区，各有用途。这在 `partitions.csv` 中定义：

```csv
# partitions.csv
# Name,   Type, SubType,  Offset,   Size, Flags
nvs,      data, nvs,      ,         0x6000,
phy_init, data, phy,      ,         0x1000,
factory,  app,  factory,  ,         3M,
storage,  data, fat,      ,         128K,
```

图示：

```
0x9000        0x9000+24KB  0x10000      0x310000      0x330000
  ├─────────────┤            ├──────────────┤─────────────┤
  │     NVS      │  phy_init │   factory    │   storage   │
  │   (24KB)     │  (4KB)    │    (3MB)     │   (128KB)   │
  │  配置/缓存    │           │   应用程序    │   FAT 文件   │
  └─────────────┘            └──────────────┘─────────────┘
```

各分区的作用：

| 分区 | 大小 | 用途 |
|------|------|------|
| **nvs** | 24KB | 非易失性存储。WiFi 配置、天气缓存、设置项、错误日志都存在这里 |
| **phy_init** | 4KB | WiFi PHY 校准数据，ESP-IDF 引导加载器自动管理 |
| **factory** | 3MB | 编译出的固件（我们的应用程序），占用最大空间 |
| **storage** | 128KB | FAT 文件系统分区，用于存储待机壁纸图片 |

其中 NVS 分区的读写由 `esp-storage` 负责，我们会在后续文章中详细讲解其存储布局。

## 6. Feature Flags 实现多屏幕适配

这个项目支持多种尺寸的电子墨水屏，通过 Cargo feature flags 实现：

```toml
# Cargo.toml
[features]
epd2in9 = []           # 2.9 寸屏幕 (296×128) — 仅驱动层适配，页面布局缺失，暂不可编译
epd2in7 = []           # 2.7 寸屏幕 (264×176)
epd4in2 = []           # 4.2 寸屏幕 (400×300)
enable_debug = []      # 启动时若 Flash 有错误日志则进入调试页
weather-openmeteo = [] # 天气数据源切换为 Open-Meteo（默认心知天气）
```

在代码中通过 `#[cfg(...)]` 条件编译来适配不同屏幕：

```rust
// src/display.rs
#[cfg(feature = "epd2in9")]
pub type EpdDisplay = Display2in9;          // 296×128

#[cfg(feature = "epd4in2")]
pub type EpdDisplay = Display4in2;          // 400×300
```

不同屏幕的 UI 布局也会不同，比如文字行数：

```rust
// src/epd2in9_txt.rs
#[cfg(feature = "epd2in9")]
const LINES_NUM: u32 = 7;     // 2.9 寸屏幕显示 7 行文字

#[cfg(feature = "epd2in7")]
const LINES_NUM: u32 = 9;     // 2.7 寸屏幕显示 9 行文字

#[cfg(feature = "epd4in2")]
const LINES_NUM: u32 = 22;    // 4.2 寸屏幕显示 22 行文字
```

构建时通过 `--features` 参数选择（均需 `--release`）：

```bash
cargo run --release --features epd4in2               # 4.2 寸版本（400×300）
cargo run --release --features epd2in7               # 2.7 寸版本（264×176）
cargo run --release --features "epd4in2,enable_debug" # 带调试模式
```

> 目前可正常构建的是 `epd4in2` 与 `epd2in7`。`epd2in9` 只在驱动层保留了类型适配，但天气页/日历页缺少 128×296 布局文件，编译会失败。

## 7. 程序入口与初始化流程

现在来看 `main()` 函数的完整初始化流程。这是整个项目的"启动引擎"：

```rust
// src/main.rs (有删减)
#[esp_rtos::main]
async fn main(spawner: Spawner) -> ! {
    // 1. 分配堆内存
    esp_alloc::heap_allocator!(size: 64 * 1024);  // 64KB（请求缓冲已放 .bss，把 SRAM 让给静态变量/栈）

    // 2. HAL 初始化
    let config = HalConfig::default().with_cpu_clock(CpuClock::max());
    let peripherals = hal_init(config);

    // 3. 配置 Embassy 定时器
    let timg0 = TimerGroup::new(peripherals.TIMG0);
    let sw_int = SoftwareInterruptControl::new(peripherals.SW_INTERRUPT);
    esp_rtos::start(timg0.timer0, sw_int.software_interrupt0);

    // 4. 初始化 RTC
    let rtc = esp_hal::rtc_cntl::Rtc::new(peripherals.LPWR);

    // 5. 加载 Flash 存储
    crate::storage::enter_process().await;

    // 6. 配置 GPIO 引脚
    let epd_busy = peripherals.GPIO6;
    let epd_cs = Output::new(peripherals.GPIO3, Level::High, OutputConfig::default());
    // ... 更多引脚

    // 7. 初始化 SPI 总线
    let spi = Spi::new(
        peripherals.SPI2,
        SpiConfig::default()
            .with_frequency(Rate::from_mhz(32))
            .with_mode(Mode::_0),
    ).unwrap()
    .with_sck(epd_sclk)
    .with_miso(epd_miso)
    .with_mosi(epd_mosi);

    // 8. SPI 总线共享给 EPD 和 SD 卡
    let shared_spi = CsMutex::new(RefCell::new(spi));
    let spi_bus_epd = CriticalSectionDevice::new_no_delay(shared_spi_static, epd_cs).unwrap();
    let spi_bus_sd = CriticalSectionDevice::new_no_delay(shared_spi_static, sdcard_cs).unwrap();

    // 9. 启动显示渲染任务
    spawner.spawn(display::render(spi_bus_epd, epd_busy, epd_rst, epd_dc)).unwrap();

    // 10. 挂载 SD 卡
    let sdcard = SdCard::new_with_options(spi_bus_sd, embassy_time::Delay, ...);
    let volume_mgr = VolumeManager::new(sdcard, TimeSource);

    // 11. 启动事件检测任务
    spawner.spawn(event::run(key1, key2)).unwrap();

    // 12. 根据是否配置过 WiFi，选择 AP 配网或 STA 连接
    if need_ap {
        crate::wifi::start_wifi_ap(&spawner, rng, peripherals.WIFI).await;
    } else {
        spawner.spawn(crate::worldtime::ntp_worker()).unwrap();
        spawner.spawn(pages::main_task(spawner.clone())).unwrap();
        crate::wifi::connect_wifi(&spawner, rng, peripherals.WIFI).await;
    }

    // 13. 主循环：显示时间 + 自动睡眠检测
    loop {
        // 显示当前时间
        // 检查是否需要进入深度睡眠
        to_sleep_tips(Duration::from_secs(0), Duration::from_secs(180), true).await;
        Timer::after(Duration::from_secs(5)).await;
    }
}
```

这个流程可以用一张图来概括：

```
esp_alloc::heap_allocator!(64KB)
        │
   hal_init(CpuClock::max())     ← 获取所有外设的所有权
        │
   esp_rtos::start(timg0, sw_int)  ← 启动 Embassy 调度器
        │
   Rtc::new()                     ← RTC 实时时钟
        │
   storage::enter_process()       ← 从 Flash 加载配置
        │
   GPIO / SPI / ADC 初始化        ← 硬件引脚配置
        │
   ┌─ spawn(display::render)      ← 显示任务（独立 Embassy task）
   │
   ├─ SdCard + VolumeManager      ← SD 卡文件系统
   │
   ├─ spawn(event::run)           ← 事件检测任务
   │
   ├─ 未配置 WiFi?
   │    ├── Yes → AP 模式配网
   │    └── No  → spawn(ntp_worker)
   │            → spawn(main_task) ← 主界面任务
   │            → connect_wifi()
   │
   └─ 主循环：时间显示 + 睡眠检测
```

### 一个关键概念：外设所有权

注意这行代码：

```rust
let peripherals = hal_init(config);
```

`peripherals` 是一个结构体，包含了 ESP32-C3 的所有硬件外设（GPIO、SPI、ADC、WiFi 等）。在 esp-hal 的设计中，**每个外设只能被获取一次**——这由 Rust 的类型系统在编译期保证。

当你写了：

```rust
let epd_busy = peripherals.GPIO6;
```

`GPIO6` 就被"移动"出了 `peripherals`，后续代码不能再访问它。硬件外设的独占访问就这样由编译器在编译期保证——所有权一旦移走就编译不过。

## 8. 内存分配器

嵌入式环境没有操作系统提供的 `malloc/free`，需要自己管理堆内存。本项目使用 `esp-alloc`：

```rust
esp_alloc::heap_allocator!(size: 64 * 1024);  // 64KB 堆内存
```

这一行宏做了三件事：
1. 在 ESP32-C3 的 SRAM 中划出 64KB 作为堆空间
2. 初始化一个全局内存分配器
3. 将其注册为 Rust 的 `GlobalAlloc`

> 为什么是 64KB 而不是更多？HTTP 请求/响应、TLS 这类大缓冲被放到了 `.bss`（静态区），由 WiFi 请求锁串行化独占使用，不必占用堆；这样能把宝贵的 SRAM 留给栈和静态变量。

```rust
extern crate alloc;        // 引入 alloc crate
use alloc::string::String;  // 现在可以使用 String 了
use alloc::vec::Vec;        // 现在可以使用 Vec 了
```

ESP32-C3 总共有约 400KB SRAM，其中 64KB 划给堆，剩余的被栈、静态变量（含放进 .bss 的网络缓冲）、Embassy 任务占用。在资源受限的嵌入式环境中，内存分配是一个需要时刻关注的问题——堆碎片化可能导致分配失败。因此在关键路径上，项目优先使用固定大小的 `heapless` 集合：

```rust
use heapless::String as HString;
let ssid: HString<32> = HString::new();  // 固定 32 字节，在栈上分配，不会碎片化
```

## 9. Edition 2024

本项目使用 Rust Edition 2024：

```toml
edition = "2024"
```

这是目前最新的 Rust 版本，带来了一些影响代码风格的变化：
- **枚举变体使用 PascalCase**：`Mode::_0` 而非 `Mode::Mode0`
- **`impl Trait` 在更多位置可用**：得益于 nightly feature
- **生命周期推断规则调整**：部分场景需要更明确的生命周期标注

## 小结

本文建立了项目的全景视图：

| 概念 | 要点 |
|------|------|
| `#![no_std]` | 嵌入式 Rust 不使用标准库，只有 `core` |
| `#![no_main]` | 由 `esp_rtos::main` 宏提供程序入口 |
| 目标平台 | `riscv32imc-unknown-none-elf`，RISC-V 裸机 |
| 工具链 | nightly Rust + espflash 烧录 |
| 分区表 | NVS(24KB) + Factory(3MB) + Storage(128KB) |
| 堆内存 | esp-alloc 提供 64KB 全局堆（大缓冲放 .bss） |
| Feature Flags | 编译期选择屏幕尺寸和数据源 |
| 外设所有权 | 编译期防止硬件外设被重复使用 |
